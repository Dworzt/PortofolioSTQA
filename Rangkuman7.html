Kelompok 7: Automation Testing (Selenium)

Panduan praktis Selenium: arsitektur, menulis tes yang maintainable (Page Object Model), strategi menunggu (waits), tips stabilitas, dan cara mengintegrasikan eksekusi ke CI/CD.

Selenium Suite — Komponen Utama

  • WebDriver: API untuk mengontrol browser (ChromeDriver, GeckoDriver).
  • Selenium Grid: menjalankan tes paralel di berbagai node/browser.
  • Selenium IDE: rekaman & generasi skrip dasar.

Page Object Model (POM)

POM memisahkan locator & interaksi ke kelas/objek halaman, sehingga tes menjadi lebih ringkas dan maintainable. Perubahan UI cukup diperbaiki di satu lokasi (page object).

  • Contoh class: LoginPage.login(username, password)
  • Prinsip: DRY (Don't Repeat Yourself) dan single responsibility untuk setiap page object.

Waits & Synchronization

Gunakan explicit waits (ExpectedConditions) untuk elemen yang muncul dinamis. Hindari implicit waits yang dapat membuat debugging sulit.

  • Explicit wait: tunggu kondisi spesifik (clickable, visible)
  • Retry strategies: coba ulang tindakan jika element stale

Selectors yang Stabil

Prefer data-* attributes (data-test, data-cy) atau id yang stabil. Hindari absolute XPaths kecuali tidak ada alternatif.

Fixture, Setup & Teardown

Pastikan setiap tes memulai dari state yang konsisten: seed data, clear cache, reset DB, atau gunakan test accounts khusus.

CI, Grid & Scaling

Jalankan smoke E2E pada setiap PR dan full suite di nightly build. Untuk skalabilitas gunakan Selenium Grid atau cloud services (BrowserStack, Sauce Labs) dan parallelization untuk mengurangi waktu run.

Mengurangi Flakiness

  • Gunakan explicit waits, retry untuk operasi network, dan deteksi kondisi asinkronis.
  • Kurangi ketergantungan antar tes—jangan membuat state bergantung pada urutan eksekusi.
  • Hindari assertions yang terlalu ketat pada UI (mis. pixel perfect) kecuali diperlukan.
