Kelompok 4: Test Scenario, Case, & Bug Report

Catatan lengkap tentang bagaimana menerjemahkan kebutuhan dan test plan menjadi scenario dan test case yang dapat dieksekusi, serta bagaimana menyusun bug report yang membantu proses perbaikan menjadi efisien.

Dari Requirement → Scenario → Test Case

Alur ideal: requirement/user story → test scenario (high-level) → test case (langkah terperinci). Scenario membantu merangkum intent; case memungkinkan reproduksi dan automasi.

  • Scenario: "Pengguna menghitung BMI dan melihat history"
  • Test Case: langkah-langkah login → input data → klik hitung → verifikasi hasil → verifikasi history tersimpan

Menyusun Test Case yang Baik (Checklist)

  • Gunakan ID unik dan judul deskriptif (TC-001: Valid BMI Calculation).
  • Isi: tujuan, pra-kondisi, langkah (sederhana & terukur), data uji, hasil yang diharapkan, pasca-kondisi.
  • Tambahkan prioritas dan estimasi waktu eksekusi.
  • Catat apakah test bisa diotomasi atau manual, dan jika otomatis, link ke skrip.

Template Singkat Test Case

ID: TC-001
Title: Valid BMI Calculation
Preconditions: User logged in, form available
Steps:
  1. Open BMI page
  2. Set height=170, weight=65
  3. Click "Hitung"
Expected: BMI = 22.5 (approx) dan entry muncul di history

Bug Report: Struktur & Contoh

Bug report harus memudahkan developer memahami, mereproduksi, dan memperbaikinya. Sertakan environment, langkah reproduksi, aktual vs ekspektasi, attachment (screenshot/log), severity & priority.

Title: Error when saving BMI history (500)
Steps: 1) Login; 2) Calculate BMI; 3) Click save
Actual: Server returns 500, entry not saved
Expected: 201 Created, entry saved in history
Env: Staging, build v1.2.0, Chrome 116
Severity: Major — data loss
Attachments: Screenshot, server logs

Triage & Alur Penanganan Bug

  1. Validasi laporan (QA reproduces)
  2. Assign & klasifikasi (severity/priority)
  3. Developer investigates & fixes
  4. QA verifies fix, regression test related flows
  5. Close bug with notes

Eksekusi Test & Hasil

Standarisasi hasil: Passed, Failed, Blocked, Skipped. Selalu hubungkan hasil ke test case ID supaya traceability terjaga.
