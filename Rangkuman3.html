Kelompok 3: Perencanaan Pengujian (Test Plan)

Panduan detail tentang menyusun Test Plan yang efektif — struktur dokumen, elemen kritis, contoh template, serta praktik terbaik untuk memastikan proses pengujian terorganisir.

Definisi & Tujuan

Test Plan adalah dokumen resmi yang menetapkan strategi pengujian, cakupan, sumber daya, jadwal, deliverables, dan kriteria keberhasilan/keluaran. Tujuannya: mengurangi ambiguitas, menyelaraskan tim, dan menyediakan referensi sepanjang fase pengujian.

Dokumen ini sebaiknya ditinjau bersama stakeholder (PM, dev lead, QA lead) sebelum eksekusi.

Struktur Test Plan (ringkasan IEEE 829)

  1. Test Plan Identifier & Revision History
  2. Introduction — tujuan dan ruang lingkup
  3. Test Items — fitur/komponen yang diuji
  4. Features to be Tested / Not to be Tested
  5. Approach — strategi pengujian & teknik
  6. Pass/Fail Criteria dan Suspension Criteria
  7. Test Deliverables (dokumen, laporan, skrip)
  8. Testing Tasks & Schedule
  9. Environmental Needs (hardware, software, test data)
  10. Responsibilities & Staffing
  11. Risks, Assumptions, & Mitigation

Contoh Ringkas Template

Test Plan ID: TP-APP-001
Project: Aplikasi BMI
Scope: Fitur kalkulasi BMI & history
Approach: Manual regression + automasi unit tests dan API smoke tests
Entry Criteria: Build passing smoke tests, staging environment ready
Exit Criteria: Semua critical bugs fixed, >90% test case pass

Pendekatan & Teknik

Pilih teknik berdasarkan risiko dan tujuan: exploratory testing, regression, smoke, sanity, performance sampling, dan security scanning.

  • Prioritaskan test case untuk area kritis dan sering dipakai.
  • Gunakan automasi untuk smoke/regression yang stabil.
  • Selenggarakan exploratory sessions untuk menemukan masalah yang tidak tertutupi test case.

Sumber Daya, Jadwal & Estimasi

Buat breakdown tasks: test design, test data preparation, environment setup, eksekusi, dan reporting. Sertakan estimasi effort (person-hours) untuk setiap aktivitas dan dependencies.

  • Contoh: Test design (16 jam), Test data (8 jam), Eksekusi (40 jam), Bug triage (8 jam).

Pass/Fail, Suspension & Resumption Criteria

Tentukan batasan untuk menilai apakah build dapat diuji atau harus ditunda. Contoh: jika >10% test case failed atau ada blocking bug, suspend dan notif tim dev untuk hotfix.

Risiko & Mitigasi

Contoh risiko: tidak tersedianya test environment, delay dependency API, kurangnya data. Mitigasi: environment fallback, mock services, parallelize tasks.
